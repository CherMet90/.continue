Цель: ведение документации для личной базы знаний.

Аудитория: mid-level инженеры/сисадмины, чтение через mkdocs.

## Структура документа
Документ должен придерживаться шаблона:

1. **Frontmatter** (YAML, в начале файла):
   - `title: Название`
   - `date: YYYY-MM-DD`

2. **Название статьи** (h1)
3. **Теоретическая часть: терминология, основы и т.п.** (h2, темы внутри можно логически разделять через h3):
   - Теоретическая часть нужна для создания сильного фундамента для понимания контекста задач.
   - Подача наглядная, структурированная. Приветствуется mermaid, если уместно.
   - Нюансы в теоретической части не нужны - оставь их для примечаний в кейсах.

4. **Практическая часть** (h2 «Название кейса»):
   - Название кейса — краткая формулировка темы задачи (например, «Настройка агрегации VLAN»).
   - **Команды** — в кодовых блоках с указанием языка (например, ```bash), строго те, что реально использовались. Не додумывай «полезных» аналогов.
     - Внутри кодовых блоков разрешены краткие inline-комментарии (`# комментарий`) только в ключевых, важных в момент применения команды местах.
   - **Примечания** — если есть важные нюансы, ограничения, подводные камни, выноси их после кодового блока отдельным параграфом или списком, используя **жирное выделение** для ключевого слова: `**Важно:** ...`, `**Ограничение:** ...` и т.п.
   - При необходимости разбивай команды внутри кейса на логические блоки, чередуй с соответствующими примечаниями
   - **Источники:**  
     Если при создании кейса использовалась внешняя документация (статьи, refdocs, man-страницы), добавь в конце кейса маркированный список с человекочитаемыми названиями и ссылками.  

## Правила наполнения
- Обезличивай: реальные имена хостов, IP, домены заменяй на обобщённые (например, `srv-app-01`, `10.0.1.10`, `example.internal`).
- Минимум воды. Документ должен быть легко «пробегаем глазами».
- Документ предполагает, что он может обновляться и дополняться новыми темами и кейсами в будущем.

Пиши так, чтобы документ успешно рендерился в mkdocs-material — используй корректную Markdown-разметку (подсветка кода, жирный текст, списки).
